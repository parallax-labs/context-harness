# RUNBOOK-0005: Cut a Release

**Status:** Active
**Date:** 2026-02-28
**Author:** pjones
**Last Verified:** 2026-02-28

## Purpose

This runbook walks through cutting a new release of Context Harness. Use it when publishing a new version (e.g. `v0.6.0`) to GitHub Releases. The release workflow (`.github/workflows/release.yml`) is triggered by pushing a tag matching `v*` and produces platform binaries, checksums, and a release blog post stub.

## Prerequisites

- Push access to the repository
- Git configured with your identity
- GitHub CLI (`gh`) installed (optional, for verifying releases)
- All intended changes merged to `main`

## Steps

1. Ensure `main` is clean and up to date.

   ```bash
   git checkout main
   git pull origin main
   git status
   ```

   Expected: Working tree clean; no uncommitted changes.

2. Update the version in `Cargo.toml` if needed. The version must match the tag you will create (e.g. `0.6.0` for tag `v0.6.0`).

   ```bash
   # Edit crates/context-harness/Cargo.toml and set version = "0.6.0"
   # If the workspace has other versioned crates, update them too.
   ```

   If you changed the version, commit and push:

   ```bash
   git add crates/context-harness/Cargo.toml
   git commit -m "chore: bump version to 0.6.0"
   git push origin main
   ```

   Expected: Commit pushed; CI passes on `main`.

3. Create and push the release tag. Use semantic versioning (e.g. `v0.6.0`).

   ```bash
   git tag v0.6.0
   git push origin v0.6.0
   ```

   Expected output (or similar):

   ```
   Enumerating objects: ...
   To github.com:parallax-labs/context-harness.git
    * [new tag]         v0.6.0 -> v0.6.0
   ```

4. Monitor the release workflow. Open the Actions tab on GitHub and watch the "Release" workflow run.

   ```bash
   gh run list --workflow=release.yml --limit 5
   ```

   Expected: Workflow run appears; all 6 build jobs complete; "Create Release" job succeeds.

5. Verify the GitHub Release page. Navigate to **Releases** → the new tag (e.g. `v0.6.0`).

   - Release notes are auto-generated by `softprops/action-gh-release`.
   - Assets include 6 platform artifacts: `.tar.gz` (Unix) and `.zip` (Windows) with `.sha256` checksums.

   Expected: Release visible; all 6 platform artifacts present; checksum files present.

6. Verify artifacts. Download one or more binaries and confirm they run.

   ```bash
   # Example: download and verify Linux x86_64
   curl -sL -o ctx-linux-x86_64.tar.gz "https://github.com/parallax-labs/context-harness/releases/download/v0.6.0/ctx-linux-x86_64.tar.gz"
   curl -sL -o ctx-linux-x86_64.tar.gz.sha256 "https://github.com/parallax-labs/context-harness/releases/download/v0.6.0/ctx-linux-x86_64.tar.gz.sha256"
   shasum -a 256 -c ctx-linux-x86_64.tar.gz.sha256
   tar xzf ctx-linux-x86_64.tar.gz
   ./ctx --help
   ```

   Expected: Checksum validates; `ctx --help` prints usage.

7. Check the blog post PR. The `blog-post` job creates a PR with a release blog stub in `site/content/blog/`.

   ```bash
   gh pr list --head "release-blog/v0.6.0" --state open
   ```

   Expected: PR titled "Blog: release notes for v0.6.0" exists. Edit the stub, fill in TODO placeholders, then merge to publish.

## Verification

- GitHub Release exists for the tag with auto-generated notes.
- All 6 platform binaries are present: `ctx-linux-x86_64`, `ctx-linux-x86_64-musl`, `ctx-linux-aarch64`, `ctx-macos-x86_64`, `ctx-macos-aarch64`, `ctx-windows-x86_64`.
- Each artifact has a matching `.sha256` checksum file.
- At least one binary runs correctly (`ctx --help`).
- Blog post PR is open (or merged) for the release.

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| Workflow fails on a specific build job | Platform-specific build error (e.g. Zig, tract, ORT) | Check the failing job logs; reproduce locally with `cargo build --release --target <target> -p context-harness` (see [RUNBOOK-0006](0006-ci-failures.md)) |
| Release created but artifacts missing | Upload/download artifact step failed | Re-run the failed job; if artifacts are corrupted, delete the release and re-push the tag (after fixing the workflow) |
| Tag already exists | Tag was pushed previously | Use `git tag -d v0.6.0` locally and `git push origin :refs/tags/v0.6.0` to delete the remote tag; fix issues, then re-tag and push |
| Blog post PR not created | `site/content/blog/` unchanged, existing post detected, or workflow change detection missed an untracked stub | Run `./scripts/generate-release-blog.sh v0.6.0` locally, check `git status --short site/content/blog/`, and open a manual PR |
| Version mismatch between tag and Cargo.toml | Tag pushed before version bump | Delete the tag, bump version, commit, push, then re-create and push the tag |
